## 1. Введение

### 1.1. Назначение документа

Настоящий документ описывает требования к мобильному приложению «Клеточный футбол» — цифровой реализации игры в стиле «футбол на бумаге» на клетчатом поле. Документ предназначен для преподавателя и разработчиков учебного проекта, задаёт границы версии 1.0 и направления развития (Phase 2). Документ является основой для проектирования интерфейса, архитектуры клиента и последующей реализации.

### 1.2. Область применения продукта

Продукт предназначен для пользователей мобильных устройств (смартфоны; при необходимости планшеты). В версии 1.0 основные сценарии не требуют постоянного сетевого взаимодействия между удалёнными игроками. Сетевой режим PvP между удалёнными пользователями вынесен в Phase 2.

### 1.3. Определения, сокращения, ссылки

**SRS** — Software Requirements Specification, спецификация требований к ПО.  
**PvP** — Player versus Player, режим «игрок против игрока».  
**ИИ** — искусственный интеллект (алгоритм соперника в приложении).  
**BaaS** — Backend as a Service, облачный backend как сервис.  

**Ссылки:**

- К. Вигерс, «Разработка требований к программному обеспечению» — методология выявления и документирования требований (учебная литература).
- IEEE / ГОСТ — ориентиры по структуре документов требований (в рамках учебной работы).
- Предметная область и бытовое описание игры: [Игра «Футбол на бумаге» (Golos)](https://golos.id/ru--igry/@sergiy/igra-futbol-na-bumage) — как вспомогательный источник, не как норматив.

## 2. Общее описание

### 2.1. Контекст и обоснование

Игра «футбол на бумаге» традиционно ведётся на клетчатом поле от руки. Цифровая версия устраняет необходимость бумаги и ручной фиксации линий, обеспечивает контроль соблюдения правил, даёт возможность играть против ИИ и вести локальные турниры. Продукт позиционируется как развлекательное и учебное приложение с чётко зафиксированными правилами v1.0.

### 2.2. Цели продукта

1. Обеспечить **корректную по правилам v1.0** партию на клетчатом поле с визуализацией линий и мяча.
2. Предоставить режим **игры против ИИ** для тренировки и игры в одиночку.
3. Предоставить режим **локального PvP** на **одном устройстве** (поочерёдные ходы).
4. Предоставить режим **ручного турнира**: создание участников, сетка, фиксация результатов матчей на стороне клиента (в объёме v1.0).
5. Заложить основу для **Phase 2**: онлайн-PvP с использованием **сетевого компонента** (облачный сервис или собственный API).

### 2.3. Заинтересованные стороны и пользователи

| Роль | Интересы |
|------|----------|
| Игрок (казуальный) | Быстрый старт партии, понятный интерфейс, соблюдение правил |
| Игрок (соревновательный) | Турниры, прозрачный счёт, отсутствие спорных трактовок правил |
| Преподаватель / проверяющий | Наличие SRS, просматриваемые на GitHub артефакты, трассируемость требований |
| Разработчик (учебный) | Ясные границы v1.0, отдельный документ правил, расширяемость под Phase 2 |

### 2.4. Ограничения и допущения

- Версия 1.0 **не включает** онлайн-PvP между удалёнными игроками.
- Точная трактовка геометрии сетки («пересечение/касание» линий) может потребовать уточнения в технической спецификации; поведение должно быть **согласовано** с [game-rules.md](game-rules.md).
- Минимальные версии ОС и целевой стек реализации **уточняются** после согласования с преподавателем (ориентир: кроссплатформенный фреймворк или нативная разработка).

### 2.5. Зависимости

- Мобильная ОС (Android и/или iOS — уточняется).
- Для v1.0: **нет обязательной** зависимости от внешнего игрового сервера.
- Для Phase 2: зависимость от **сетевого компонента** (облачный сервис или собственный API).

## 3. Границы системы и версионирование

### 3.1. Версия 1.0 — объём поставки

В v1.0 входит клиентское мобильное приложение, реализующее:

- игровое поле и механику согласно [game-rules.md](game-rules.md);
- режим **против ИИ**;
- режим **локального PvP** на одном устройстве;
- режим **ручного турнира** (создание участников, сетка, ввод результатов);
- базовые экраны: главное меню, выбор режима, игровой экран, экран правил/справки (с отсылкой к game-rules), настройки при необходимости.

### 3.2. Вне объёма версии 1.0

- Онлайн-PvP в реальном времени между удалёнными устройствами.
- Глобальные рейтинги, учётные записи с обязательной регистрацией (если не требуется явно в v1.0).
- Монетизация, реклама, магазин контента.

### 3.3. Phase 2 — планы развития

В Phase 2 планируется **онлайн-PvP**: подбор соперника или приглашение, синхронизация состояния партии, идентификация игроков, восстановление сессии.

Для Phase 2 потребуется **сетевой компонент**: использование **облачного сервиса** (например, класса BaaS) **или** развёртывание **собственного API** (REST/WebSocket) с серверной логикой, обеспечивающей **источник истины** для состояния партии.

## 4. Предметная область и аналоги

### 4.1. Описание предметной области

Игра ведётся на прямоугольном клетчатом поле. Игроки по очереди выполняют ходы фиксированной длины, строят линии на сетке, стремятся забить гол в ворота соперника. Правила окончания партии, штанги, штрафные удары, специальные фазы после исчерпания точек продолжения и т.д. — **строго по** [game-rules.md](game-rules.md).

### 4.3. Отличия и уникальные особенности проекта

- Явно описанные **границы v1.0** и отложенный **онлайн-PvP (Phase 2)**.
- Учёт **ручных турниров** в объёме учебного проекта.

## 5. Правила игры (нормативная отсылка)

Полный нормативный текст правил: **[game-rules.md](game-rules.md)** (v1.0).

**Краткое резюме (не заменяет game-rules.md):** поле 20×28, ворота шириной 6 на короткой стороне; обычный ход — ломаная из трёх клеток; ограничения на пересечение/касание линий; гол и ввод после гола с линии центра; штанги и отскоки; штрафной 7 клеток; фаза «7 ходов» и ничья 0–0 при описанных условиях; завершение матча при невозможности продолжения после гола.

## 6. Внешние интерфейсы

### 6.1. Пользовательские интерфейсы

Интерфейс сенсорный; отображение поля, линий, мяча, текущего игрока, счёта; экраны согласно мокапам в каталоге `/mockups/` репозитория.

- **Phase 2:** сетевой обмен через выбранный сетевой компонент (см. раздел 3.3).

## 7. Функциональные требования

### 7.1. Общие функции приложения

| ID | Требование |
|----|------------|
| FR-APP-01 | Пользователь может просмотреть краткие правила и перейти к полному тексту (отсылка на game-rules или встроенная копия). |
| FR-APP-02 | Пользователь может начать новую партию из главного меню, выбрав режим (ИИ / локальный PvP / турнир). |
| FR-APP-03 | Приложение отображает текущее состояние партии: положение мяча, нарисованные линии, счёт (если применимо). |
| FR-APP-04 | Пользователь может прервать партию (выход в меню); поведение счёта и автосохранения незавершённой партии задаётся в реализации и согласуется с NFR-03. |

### 7.2. Режим «Игра против ИИ»

| ID | Требование |
|----|------------|
| FR-AI-01 | Пользователь может начать партию против ИИ. |
| FR-AI-02 | ИИ выполняет только **допустимые** ходы согласно [game-rules.md](game-rules.md). |
| FR-AI-03 | Пользователь может завершить партию досрочно (выход в меню) — см. FR-APP-04. |

### 7.3. Режим «Локальный PvP» (одно устройство)

| ID | Требование |
|----|------------|
| FR-PVP-01 | Два игрока могут играть по очереди на одном устройстве. |
| FR-PVP-02 | Интерфейс явно показывает, чей сейчас ход. |
| FR-PVP-03 | Недопустимые ходы отклоняются с понятным сообщением. |

### 7.4. Режим «Турнир (ручное ведение)»

| ID | Требование |
|----|------------|
| FR-TRN-01 | Пользователь может создать турнир и добавить список участников (имена/никнеймы). |
| FR-TRN-02 | Пользователь может задать формат сетки (**олимпийка / на выбывание** — **уточнить и зафиксировать**). |
| FR-TRN-03 | Пользователь может вручную фиксировать результат матча (победитель / счёт — в объёме, согласованном с правилами турнира). |

### 7.5. Игровой движок и соответствие правилам

| ID | Требование |
|----|------------|
| FR-ENG-01 | Все исходы хода, гола, штанги, штрафного, фаз «7 ходов», завершения матча реализуются в соответствии с [game-rules.md](game-rules.md). |
| FR-ENG-02 | Система не позволяет завершить обычный ход, нарушающий ограничения разделов 4–5 game-rules. |

### 7.6. Phase 2 — онлайн-PvP (план)

| ID | Требование |
|----|------------|
| FR-NET-01 | Реализовать онлайн-PvP с синхронизацией партии через сетевой компонент (облачный сервис или собственный API). |

## 8. Нефункциональные требования

| ID | Требование |
|----|------------|
| NFR-01 | Отклик интерфейса на действия пользователя: без заметных задержек при типичном ходе (целевое значение уточняется). |
| NFR-02 | Корректная работа на целевых версиях ОС (уточняются). |
| NFR-03 | Данные локальных турниров и незавершённой партии: поведение при выходе из приложения — **сохранять / не сохранять** — **выбрать и описать** в следующей версии SRS или в проектных решениях. |
| NFR-04 | Минимизация персональных данных в v1.0 (при отсутствии аккаунтов — не собирать). |

## 9. Сценарии использования

### 9.1. Актёры

Игрок; организатор турнира (может совпадать с игроком); в Phase 2 — удалённый соперник.

### 9.2. Примеры сценариев

**UC-01. Партия против ИИ**  
Предусловия: установлено приложение.  
Основной поток: главное меню → режим ИИ → выполнение ходов до завершения по правилам.

**UC-02. Локальный PvP**  
Предусловия: два игрока у одного устройства.  
Основной поток: выбор режима → поочерёдные ходы → завершение.

**UC-03. Турнир**  
Предусловия: пользователь создаёт турнир.  
Основной поток: ввод участников → генерация/отображение сетки → ввод результатов матчей.
