Документ описывает **потоки событий** (Flow of events) для вариантов использования, отражённых на диаграмме **Use Case** (страница `03-use-case` в `cell-football-lr34.drawio`).

## UC-RULES-01. Просмотреть правила игры

| Поле | Содержание |
|------|------------|
| **ID** | UC-RULES-01 |
| **Название** | Просмотреть правила игры |
| **Актёры** | **Игрок** (основной). **Система** (мобильное приложение). |
| **Цель** | Ознакомиться с правилами v1.0 перед игрой или во время использования приложения. |
| **Предусловия** | Приложение установлено и запущено. Отображается экран, из которого доступен переход к правилам (например, главное меню). |
| **Постусловия (успех)** | Пользователь просмотрел краткие правила и/или открыл полный текст; возврат в предыдущий экран возможен. |
| **Трассировка SRS** | FR-APP-01 |

**Основной поток**

1. Игрок находится на экране, согласованном с макетом главного меню ([`mockups/01-glavnaya.png`](../../mockups/01-glavnaya.png)) или эквивалентном экране навигации v1.0.
2. Игрок выбирает пункт интерфейса, соответствующий **«Правила»** / **«Справка»** (точная подпись определяется макетом).
3. Система отображает экран **кратких правил** (текст, прокрутка при необходимости).
4. Система отображает элемент **«Полный текст правил»** или аналог (ссылка/кнопка).
5. Игрок активирует переход к полному тексту.
6. Система отображает полный текст из [game-rules.md](../game-rules.md) **или** встроенную копию того же содержания (веб-представление, встроенный просмотрщик и т.д. — на этапе реализации).
7. Игрок прокручивает и читает документ.
8. Игрок выбирает **«Назад»**.
9. Система закрывает экран правил и возвращает на экран, с которого был выполнен переход (обычно главное меню).

**Альтернативные потоки**

- **A1 (шаг 3):** Игрок сразу нажимает **«Назад»**, не открывая полный текст. Система возвращает на предыдущий экран.

**Потоки ошибок**

- **E1 (шаг 6):** Полный текст недоступен (файл не найден, ошибка встроенного просмотра). Система показывает сообщение об ошибке и предлагает повторить или вернуться назад.

## UC-MODE-01. Выбрать режим игры

| Поле | Содержание |
|------|------------|
| **ID** | UC-MODE-01 |
| **Название** | Выбрать режим игры |
| **Актёры** | **Игрок**. **Система**. |
| **Цель** | Выбрать один из режимов: против ИИ, локальный PvP на одном устройстве, турнир (ручное ведение). |
| **Предусловия** | Приложение запущено. Открыт экран выбора режима ([`mockups/02-rezhim.png`](../../mockups/02-rezhim.png)) либо экран с тем же смыслом в реализации. |
| **Постусловия (успех)** | Выбранный режим зафиксирован системой; выполнен переход к соответствующему сценарию (партия или турнир). |
| **Трассировка SRS** | FR-APP-02 |

**Основной поток**

1. Система отображает экран выбора режима ([`02-rezhim.png`](../../mockups/02-rezhim.png)).
2. Система отображает доступные в v1.0 варианты: **«Против ИИ»**, **«Два игрока»** (локальный PvP), **«Турнир»** (или эквивалентные подписи).
3. Игрок выбирает один из вариантов.
4. Система сохраняет выбранный **режим** в контексте приложения (сессия/состояние навигации).
5. Система выполняет переход:
   - при выборе **ИИ** — к сценарию подготовки/старта партии против ИИ (см. UC-AI-01);
   - при выборе **PvP** — к сценарию локальной партии (см. UC-PVP-01);
   - при выборе **турнира** — к сценарию турнира (см. UC-TRN-01).
6. Система отображает соответствующий следующий экран.

**Альтернативные потоки**

- **A1 (шаг 3):** Игрок нажимает **«Назад»**. Система возвращает на главное меню ([`01-glavnaya.png`](../../mockups/01-glavnaya.png)).

**Потоки ошибок**

- **E1:** Внутренняя ошибка при сохранении выбора. Система показывает сообщение; состояние экрана не меняется или выполняется безопасный откат.

## UC-MATCH-01. Начать партию

| Поле | Содержание |
|------|------------|
| **ID** | UC-MATCH-01 |
| **Название** | Начать партию |
| **Актёры** | **Игрок**. **Система**. |
| **Цель** | Инициировать новую партию в выбранном режиме (после выбора режима). |
| **Предусловия** | Режим игры выбран (UC-MODE-01) **или** пользователь явно запускает «Новая партия» из меню (если так предусмотрено интерфейсом). |
| **Постусловия (успех)** | Создано начальное состояние партии по [game-rules.md](../game-rules.md) (размер поля, старт мяча и т.д.); отображён игровой экран. |
| **Трассировка SRS** | FR-APP-02, FR-ENG-01; связь **include** с UC-MODE-01 на диаграмме Use Case |

**Основной поток**

1. Игрок подтверждает действие **«Играть»** / **«Начать»** (или аналог) на экране, следующем после выбора режима.
2. Система создаёт объект **партии** в начальном состоянии согласно разделу **«Старт партии»** в [game-rules.md](../game-rules.md).
3. Система инициализирует **поле** 20×28, **ворота**, пустую **линию игры**, позицию **мяча** в стартовой точке.
4. Система устанавливает **очередь хода** в соответствии с режимом (ИИ / PvP).
5. Система отображает **игровой экран**: поле, мяч, индикатор текущего игрока (для PvP — явно, FR-PVP-02).
6. Система переводит приложение в режим ожидания **первого хода** текущего игрока.

**Альтернативные потоки**

- **A1:** Игрок отменяет старт (если есть диалог подтверждения). Система возвращает на экран выбора режима или главное меню.

**Потоки ошибок**

- **E1:** Не удалось создать партию (внутренняя ошибка). Система показывает сообщение и предлагает повторить или выйти в меню.

## UC-AI-01. Играть против ИИ

| Поле | Содержание |
|------|------------|
| **ID** | UC-AI-01 |
| **Название** | Играть против ИИ |
| **Актёры** | **Игрок**. **Система** (включая модуль ИИ). |
| **Цель** | Провести партию против алгоритмического соперника с соблюдением правил v1.0. |
| **Предусловия** | Выбран режим «против ИИ», партия создана (UC-MATCH-01). |
| **Постусловия (успех)** | Партия завершена по правилам или прервана пользователем (UC-EXIT-01); счёт и итог отображены согласно исходу. |
| **Трассировка SRS** | FR-AI-01, FR-AI-02, FR-AI-03, FR-APP-03, FR-ENG-01, FR-ENG-02, FR-PVP-03 (для хода человека) |

**Основной поток**

1. Система отображает игровой экран (поле, линии, мяч, счёт).
2. Система показывает индикатор: **«Ваш ход»** (или эквивалент).
3. Игрок выполняет жесты ввода **траектории обычного хода** на поле (касания по узлам/клеткам — конкретика GUI на этапе реализации).
4. Игрок подтверждает ход (если в интерфейсе есть отдельное подтверждение; иначе шаг 3 сразу отправляет ход).
5. Система проверяет ход на соответствие разделам **«Обычный ход»** и границе поля в [game-rules.md](../game-rules.md) (FR-ENG-02).
6. Система применяет допустимый ход: обновляет **линию игры**, положение **мяча**, при необходимости счёт (гол) и специальные фазы по правилам (штанга, штрафной, фаза «7 ходов» и т.д. — без переписывания всего документа здесь).
7. Система обновляет графику на экране (FR-APP-03).
8. Система проверяет условия **окончания партии** по правилам.
9. Если партия **не окончена** и очередь **человека** — возврат к шагу 2.
10. Если очередь **ИИ**, система показывает **«Ход ИИ»** (или аналог).
11. Система вычисляет **допустимый** ход ИИ (FR-AI-02).
12. Система применяет ход ИИ и обновляет экран (шаги 6–8 для стороны ИИ).
13. Цикл шагов 2–12 повторяется до завершения партии по правилам.
14. Система отображает экран **итога партии** (победа/поражение/ничья, счёт).
15. Игрок возвращается в меню (**«В меню»** / **«Назад»**).

**Альтернативные потоки**

- **A1 (шаг 5):** Ход недопустим. Система **не** меняет состояние партии; показывает **понятное сообщение** (FR-PVP-03 по смыслу для пользовательского ввода). Возврат к шагу 3.
- **A2:** Игрок вызывает **прерывание партии** во время игры. См. **UC-EXIT-01**.

**Потоки ошибок**

- **E1:** Сбой при расчёте хода ИИ. Система сообщает об ошибке; предлагает повторить ход ИИ или завершить партию без сохранения — конкретика на этапе реализации.

## UC-PVP-01. Играть локальный PvP

| Поле | Содержание |
|------|------------|
| **ID** | UC-PVP-01 |
| **Название** | Играть локальный PvP |
| **Актёры** | **Игрок 1**, **Игрок 2** (на одном устройстве; в документе можно обозначать как «текущий игрок»). **Система**. |
| **Цель** | Провести партию в режиме двух игроков на одном устройстве с поочерёдными ходами. |
| **Предусловия** | Выбран режим локального PvP; партия создана (UC-MATCH-01). |
| **Постусловия (успех)** | Партия завершена по правилам или прервана (UC-EXIT-01). |
| **Трассировка SRS** | FR-PVP-01, FR-PVP-02, FR-PVP-03, FR-APP-03, FR-ENG-01, FR-ENG-02 |

**Основной поток**

1. Система отображает игровой экран.
2. Система явно показывает, **чей сейчас ход** (FR-PVP-02), например «Ход: Игрок A» / «Ход: Игрок B».
3. Текущий игрок вводит траекторию хода (касания по полю).
4. Текущий игрок подтверждает ход (если предусмотрено).
5. Система проверяет ход по [game-rules.md](../game-rules.md) (FR-ENG-02).
6. Система применяет ход и обновляет отображение (FR-APP-03).
7. Система передаёт очередь другому игроку.
8. Система проверяет окончание партии.
9. Если партия не окончена — возврат к шагу 2 с обновлённым индикатором хода.
10. После окончания — экран итога; любой из игроков может вернуться в меню.

**Альтернативные потоки**

- **A1 (шаг 5):** Недопустимый ход — сообщение, возврат к шагу 3 (FR-PVP-03).
- **A2:** Прерывание партии — **UC-EXIT-01**.

**Потоки ошибок**

- **E1:** Конфликт ввода (одновременные касания) — на этапе реализации: игнорировать второе касание или блокировать до завершения хода.

## UC-EXIT-01. Прервать партию

| Поле | Содержание |
|------|------------|
| **ID** | UC-EXIT-01 |
| **Название** | Прервать партию |
| **Актёры** | **Игрок** (или активный игрок в PvP). **Система**. |
| **Цель** | Выйти из текущей партии в меню; при необходимости сохранить состояние локально. |
| **Предусловия** | Партия в состоянии «идёт» или «пауза»; доступен элемент «Меню» / «Выйти». |
| **Постусловия (успех)** | Партия не продолжается; отображено главное меню; при политике NFR-03 данные сохранены локально. |
| **Трассировка SRS** | FR-APP-04, NFR-03 |

**Основной поток**

1. Игрок на игровом экране выбирает **«Меню»** / **«Выйти из партии»** (или аналог).
2. (Опционально) Система показывает диалог подтверждения.
3. Игрок подтверждает выход.
4. Система **сохраняет локально** состояние незавершённой партии и связанные данные согласно **NFR-03** (если выбрана политика автосохранения при штатном выходе).
5. Система освобождает UI от игрового режима.
6. Система отображает **главное меню** ([`01-glavnaya.png`](../../mockups/01-glavnaya.png)).

**Альтернативные потоки**

- **A1 (шаг 2):** Игрок отменяет диалог. Система возвращает на игровой экран без изменений.

**Потоки ошибок**

- **E1 (шаг 4):** Ошибка записи на устройство. Система сообщает, что сохранение не выполнено; предлагает повторить выход или остаться в партии.

## UC-TRN-01. Создать турнир и сетку

| Поле | Содержание |
|------|------------|
| **ID** | UC-TRN-01 |
| **Название** | Создать турнир и сетку |
| **Актёры** | **Игрок** (организатор). **Система**. |
| **Цель** | Создать турнир, добавить участников, выбрать формат и получить сетку. |
| **Предусловия** | Приложение запущено; выбран режим турнира с экрана [`02-rezhim.png`](../../mockups/02-rezhim.png) или эквивалента. |
| **Постусловия (успех)** | В системе существует объект турнира со списком участников, форматом и отображаемой сеткой. |
| **Трассировка SRS** | FR-TRN-01, FR-TRN-02 |

**Основной поток**

1. Система отображает экран **мастера турнира** (название турнира — опционально).
2. Игрок вводит **имя первого участника** (никнейм).
3. Система добавляет участника в список.
4. Игрок повторяет ввод для остальных участников (минимум 2; верхний предел — по реализации).
5. Игрок завершает ввод списка (кнопка **«Готово»** / **«Далее»**).
6. Система предлагает выбрать **формат сетки**: **олимпийка (круговая)** или **плей-офф на выбывание** (FR-TRN-02).
7. Игрок выбирает формат.
8. Система **генерирует сетку** в соответствии с форматом и отображает её на экране.
9. Игрок просматривает пары/расписание матчей турнира.

**Альтернативные потоки**

- **A1 (шаг 4):** Игрок удаляет ошибочно введённого участника — система обновляет список.
- **A2:** Игрок нажимает **«Назад»** до сохранения турнира — система отменяет черновик или спрашивает подтверждение (на этапе реализации).

**Потоки ошибок**

- **E1:** Меньше двух участников при попытке перейти к шагу 6. Система показывает сообщение и остаётся на вводе участников.
- **E2:** Нечётное число участников для плей-офф — поведение (bye, перерасчёт сетки) задаётся реализацией; система сообщает пользователю выбранную политику.

## UC-TRN-02. Зафиксировать результат матча турнира

| Поле | Содержание |
|------|------------|
| **ID** | UC-TRN-02 |
| **Название** | Зафиксировать результат матча турнира |
| **Актёры** | **Игрок** (организатор). **Система**. |
| **Цель** | Вручную ввести исход матча турнира и обновить сетку/таблицу. |
| **Предусловия** | Турнир создан, сетка отображается; выбран конкретный матч без зафиксированного результата. |
| **Постусловия (успех)** | Результат матча сохранён в объекте турнира; сетка/статусы обновлены. |
| **Трассировка SRS** | FR-TRN-03, NFR-03 (локальное хранение турнира) |

**Основной поток**

1. Игрок на экране сетки выбирает **матч** (пара участников).
2. Система открывает форму ввода результата (победитель / счёт — в объёме, принятом в реализации).
3. Игрок вводит результат.
4. Игрок подтверждает **«Сохранить»**.
5. Система проверяет корректность ввода (непустой победитель, допустимый формат счёта и т.д.).
6. Система записывает результат в модель **«Матч турнира»**.
7. Система обновляет отображение сетки (статус «сыгран», подсветка следующего матча и т.п.).
8. Система сохраняет данные турнира **локально** (NFR-03).

**Альтернативные потоки**

- **A1 (шаг 4):** Игрок нажимает **«Отмена»**. Система закрывает форму без изменений.

**Потоки ошибок**

- **E1 (шаг 5):** Некорректный ввод. Система показывает сообщение; остаётся на шаге 3.
- **E2 (шаг 8):** Ошибка сохранения на устройство. Система предупреждает; результат может остаться только в памяти до успешного сохранения (политика на реализации).

## Примечания для согласования с преподавателем

- Связь **«Начать партию»** **include** **«Выбрать режим игры»** на диаграмме отражена в сценариях: сначала выбор режима (UC-MODE-01), затем старт партии (UC-MATCH-01).
- Детализация **игрового экрана** (макет поля) может быть расширена отдельным мокапом позже; в сценариях используется обобщённое название **«игровой экран»**.
- Политика **прерывания партии** и **сохранения** (шаги UC-EXIT-01) должна быть **одной** в реализации и совпадать с **NFR-03** в SRS.
